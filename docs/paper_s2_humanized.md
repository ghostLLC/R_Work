# 第2章 数据背景与预处理

## 2.1 TCGA数据库与BRCA数据集概述

### 2.1.1 TCGA项目简介

TCGA（The Cancer Genome Atlas）是美国NCI和NHGRI在2006年联合启动的一项癌症基因组研究计划。项目持续了十多年，覆盖33种癌症类型、超过20,000例肿瘤样本和配对正常组织，系统生成了基因组、转录组、表观基因组和蛋白质组四个层次的数据。到现在为止，TCGA依然是癌症生物信息学领域最常用的公共数据源——过去十年相当一部分重要的癌症基因组学发现，都在不同程度上受益于这个项目攒下的数据。

本研究用的BRCA队列是TCGA最早启动的癌种之一，也是样本量最大的实体瘤项目。它涵盖了Luminal A、Luminal B、HER2-enriched和三阴性四种分子亚型，分期从I到IV全覆盖。选这个队列的理由其实比较简单：样本量够大（统计学上有足够的power），临床信息够丰富（94个原始变量），而且分子分型体系成熟——对于比较不同分析方法的有效性来说，是一个相对理想的数据基础。

### 2.1.2 TCGA数据层级与条形码体系

TCGA对数据进行了严格的分层管理。从原始文件到可以直接分析的高级数据，大致分四个级别：

- **Level 1**：原始测序数据（FASTQ/BAM），仅限受控访问。
- **Level 2**：初步处理后的中间数据。
- **Level 3**：经过标准化和质控的成品数据——本研究使用的表达矩阵和突变注释都属于这个级别。
- **Level 4**：跨样本汇总的综合结果。

TCGA的样本条形码是整个数据组织方式的钥匙。以`TCGA-A8-A079-01A-11R-A002-07`为例：

| 字段位置 | 示例值 | 含义 |
|---------|--------|------|
| 项目代码 | TCGA | 项目名称 |
| 组织来源位点 | A8 | 样本采集机构编号 |
| 参与者编号 | A079 | 唯一患者标识符 |
| 样本类型 | 01 | 01=原发实体瘤，11=癌旁正常组织 |
| 小瓶标识 | A | 样本分装顺序 |
| 分析物类型 | 11 | R=RNA，D=DNA |
| 测序技术平台 | R | 平台代码 |
| 分析批次 | A002 | 实验批次编号 |
| 分析项目 | 07 | 数据加工项目编号 |

前12个字符（`TCGA-A8-A079`）就是患者级别的唯一标识，是多组学整合的锚点。跨组学对齐的时候，就是靠这12个字符把所有数据层的样本串到一起的。

### 2.1.3 BRCA多组学数据架构

本研究涉及四个数据层：

1. **mRNA转录组**（HiSeq RNA-seq）：覆盖全部基因的表达全貌，是差异分析和通路活性评估的核心输入。
2. **miRNA转录组**（HiSeq miRNA-seq）：聚焦微小RNA表达，提供转录后调控层面的信息。
3. **体细胞突变**（WES，MuTect2变异检测）：点突变、小片段插入缺失——肿瘤基因组层面的变异事件。
4. **临床信息**（TCGA Clinical）：人口统计学、病理诊断、TNM分期、分子分型、治疗方案和生存随访。

四层数据各自捕捉生物系统的不同维度——转录组看表达异常，突变看基因组层面的根源，miRNA补充表观调控信息，临床数据给所有分子发现提供表型锚点。合在一起才可能拼出一个相对完整的BRCA分子图谱。

## 2.2 数据获取与整合

### 2.2.1 基于TCGAbiolinks的数据获取流程

TCGAbiolinks是Bioconductor生态中专门用于TCGA GDC数据查询、下载和预处理的R包。它封装了GDC的RESTful API，从检索到导入一条龙。本研究的所有多组学数据都是通过它统一获取的（GDC Data Release 42.0）。

数据获取走的是"查询→下载→导入"三阶段流水线：

**第一阶段：查询（Query）**

用GDCquery构建查询对象，指定项目和数据类型：

- mRNA表达：`data.category = "Transcriptome Profiling"`，`workflow.type = "STAR - Counts"`
- miRNA表达：`data.category = "Transcriptome Profiling"`，`experimental.strategy = "miRNA-Seq"`
- 体细胞突变：`data.category = "Simple Nucleotide Variation"`，`workflow.type = "MuTect2 Variant Aggregation and Masking"`
- 临床数据：`data.category = "Clinical"`，`data.type = "Clinical Supplement"`

**第二阶段：下载（Download）**

GDCdownload批量下载，内置断点续传和自动重试——实际跑下来网络稳定性整体还行，几次chunk重试都是自动完成的。

**第三阶段：导入与解析（Prepare）**

GDCprepare把原始数据解析为R中的结构化对象。表达数据生成SummarizedExperiment（有些情况下降级为data.frame），突变数据导入为MAF格式。

### 2.2.2 多组学数据患者级整合

整合的核心操作很简单——每个样本条形码截前12个字符得到患者ID，按取交集的方式对齐。

具体步骤：
1. 从mRNA表达矩阵提取样本列名，截前12字符作为患者ID，去重。
2. 对miRNA做同样操作。
3. 突变数据提取Tumor_Sample_Barcode的前12字符。
4. 临床数据提取submitter_id的前12字符。
5. 取交集确定核心患者群体。

结果：mRNA覆盖1,094例肿瘤（另加113例正常），miRNA覆盖1,079例，两者交集1,076例，重叠率98.4%。突变数据（990例，对应983例患者）和临床数据（1,174例）进一步纳入后，最终多组学整合队列锁定在1,076例——每个患者都有完整的mRNA和miRNA表达谱外加临床注释。

## 2.3 数据预处理

### 2.3.1 mRNA表达数据的过滤

原始计数矩阵有60,660个基因（Ensembl ID），包含蛋白质编码基因、lncRNA、假基因和各种小RNA基因。直接拿60,660维的矩阵跑下游分析，噪声太大。

过滤标准很直接：基因至少在10%的样本中counts≥10——这个阈值参考了DESeq2和edgeR推荐的独立过滤准则。结果保留25,981个基因（42.8%），被滤掉的以低丰度lncRNA和假基因为主。删掉这些噪声信号对提高统计检验效能有好处——多重假设检验校正的负担会轻不少。

### 2.3.2 表达数据的归一化：TPM转换

原始Read Counts受基因长度和测序深度双重影响，不适合直接跨基因比较。TPM（Transcripts Per Million）是RNA-seq领域的标准归一化方案。

计算分两步：先做CPM（Counts Per Million）校正测序深度，再除以基因的Effective Length（以kb为单位）校正基因长度。TPM的优点是——所有基因的TPM值在每个样本内加起来是相等的，TPM值与mRNA分子的相对丰度成正比，所以跨样本比较是有意义的。

基因长度通过Ensembl基因组注释数据库（GRCh38/hg38）获取。对于极少数在最新Ensembl版本中已退役的基因ID，以对应记录进行映射和长度赋值。实际情况中Ensembl FTP的下载失败过一次（404），fallback方案是将所有基因长度统一设为1,000 bp的中位数——这不是最优解，但对后续分析的影响在可接受范围内。

### 2.3.3 基因标识符映射：Ensembl ID → Gene Symbol

TCGAbiolinks返回的是Ensembl Gene ID（如ENSG00000XXXXXXXXX）。Ensembl ID的好处是唯一且版本稳定，坏处是读起来毫无信息量——做生物学解读的时候对着"ENSG00000167208"远不如"TP53"直观。所以做了一步映射，通过org.Hs.eg.db建立Ensembl ID到Gene Symbol的对应关系。

对一对多映射（同一个Gene Symbol对应多个Ensembl ID），取平均表达量最高的那个。对完全映射不上的ID，保留原始的Ensembl ID作为后备标识。

### 2.3.4 临床数据的清洗与加工

原始临床数据94个变量，覆盖了从人口统计学到治疗随访的各个维度。从中提取了25个核心变量：

- 基本信息：确诊年龄、性别、种族
- 病理特征：AJCC分期、TNM分期、组织学类型
- 分子分型：ER、PR、HER2状态、分子亚型
- 治疗信息：既往恶性肿瘤史、放疗史
- 生存信息：生存状态、随访时间、死亡时间

有个值得一提的事情：原始分子亚型注释里发现了分类不一致的问题。具体来说是代码中的逻辑运算符优先级导致的——`ER+ | PR+ & HER2-` 被R解析为 `ER+ | (PR+ & HER2-)` 而不是正确的 `(ER+ | PR+) & HER2-`，导致部分ER+/HER2+的患者被错误归入Luminal A。修正后的分布为：Luminal A 950例（86.3%）、HER2-enriched 37例（3.4%）、Triple Negative 115例（10.4%）。Luminal A的比例偏高一点，可能跟TCGA BRCA队列入组时的选择偏倚有关。另外Luminal B最终只保留了3例（被过滤掉），原因是严格按照临床IHC指标分类后，这个亚型的样本确实少。

分期分布方面：Stage I 183例（16.6%）、Stage II 651例（58.9%）、Stage III 251例（22.7%）、Stage IV 20例（1.8%）。Stage II占比近六成，符合乳腺癌的典型分期分布。

### 2.3.5 缺失值处理

临床数据里有不同程度的缺失——部分患者某些指标没测、部分中心数据上报不完整。填补策略按变量类型分开处理：数值型用中位数（比均值抗异常值扰动），分类型用众数，生存分析中vital_status="Alive"对应的缺失days_to_death属于天然删失，不做填补。

### 2.3.6 miRNA与突变数据的预处理

miRNA方面，1,881个miRNA全部保留——未做额外的低表达过滤。理由是miRNA的reads通常比较集中，而且低丰度的miRNA可能具有关键调控功能，贸然过滤掉风险大于收益。

突变数据用的是TCGA官方推荐的MuTect2管道输出的Masked Somatic Mutation，已经过了QC和种系变异过滤。统计下来15,413个基因在至少1例患者中有体细胞突变，前20高频基因（TP53、PIK3CA、GATA3等）和已知的BRCA驱动基因谱高度吻合。

## 本章小结

这一章把数据基础搭好了。从TCGA GDC下载了四层数据（mRNA、miRNA、突变、临床），通过患者ID对齐把所有层串在了一起——最终得到一个1,076例的多组学整合队列。预处理方面，低表达过滤砍掉了约57%的基因、TPM归一化解决了测序深度和基因长度的偏倚、临床数据做了清洗和缺失值填补。比较值得一提的是分子亚型分类的那个运算符优先级bug——修完之后亚型分布更合理了，但这个坑说明即便是标准化流程中看似自动化的步骤，也值得人工检查一遍。下面的第3章，所有的数据挖掘算法都基于这套处理好的数据来跑。
